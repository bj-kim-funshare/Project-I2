# data-craft FE 위젯 시스템 재설계 — 설계 명세 (SPEC)

> 작성일: 2026-06-26 | 대상: data-craft FE 코어 재설계(Roadmap-16) — 위젯/렌더링/페이지 시스템 전면 재설계
> 상태: **설계 확정**(척추 결정 완료). 본 문서가 Roadmap-16 FE 재개편의 단일 출처(spec). 로드맵(프롬프트 시퀀스)은 본 spec을 참조해 별도 재작성.
>
> 핵심 원칙: **데이터는 순수 데이터(5 데이터타입), 표시는 블록(렌더링 타입)이 결정 — 데이터 ⊥ 표현 ⊥ 레이아웃.**

---

## 0. 확정 결정 요약 (척추)

1. **데이터 타입 = 5** (string/number/date/boolean/JSON) — 순수·불변.
2. **렌더링 타입(블록) = 41** (문자 10·버튼 12·특수 13·확장 6). "기능" 표현을 "블록"으로 대체.
3. **블록타입 = 열당 1개 고정**, `data_column.블록타입`(Not Null·41 enum)에 단일 저장. 모든 위젯이 이 단일값을 읽어 렌더 → 일관성 자동.
4. **열 타입 변경 = 별도 마이그레이션 기능**(뷰어의 기존 "열 마이그레이션" 참고 = 기존 `ColumnTypeChangeDialog`/`listTargetTypes`). 블록타입을 바꾸면 **그 열을 쓰는 모든 위젯이 자동으로 새 타입으로 전환**(데이터는 무변경, 같은 데이터타입 내 전환 시; 데이터타입을 넘으면 타입충실 데이터 변환 동반).
5. **위젯 설정 = 단일 모달**(호버→설정). 기존 인라인 디자인모드 설정 + 위젯 설정 드로어 항목 전부 이 모달로 이동.
6. **모달 3탭**: 위젯 설정(기능/데이터·블록타입) · 위젯 디자인(위젯 자체 시각) · 배경 디자인(격자).
7. **배경 디자인 = 위젯별**(공유 아님). 각 위젯이 가진 격자 내부에만 적용.

## A. 위젯 타입 (마스터 요구 원문)

1. **블록 위젯** (기존 입력 위젯을 대규모 개편하고 동작 방식을 완전히 재설계)
   - 렌더링 타입 41 종류 지원 → 데이터 그룹과 그룹 내부의 행, 열을 지정해서 특정 밸류를 타겟팅하고 해당 밸류에 대한 조회 및 쓰기를 하는 위젯 (렌더링 방식은 위젯 설정으로 관리)
   - 기존 문서 위젯을 제거하고, 블록 위젯으로 통합 흡수 (이렇게 해야 문서를 뷰어 소속인 하나의 셀로도 보고, 독립 위젯으로도 볼 수 있음, 규격 통일)
2. **디자인 위젯** (신규 디자인 2종류 추가): 텍스트 · 이미지(GIF 포함, 기업 내 공용 파일 그룹 사용) · 동영상(기업 내 공용 파일 그룹 사용) · 탭 · 바로가기(버튼, 기존 버튼을 분해)
3. **자동화 위젯**: 불러오기(버튼, 기존 버튼을 분해)
4. **보드 위젯** (신규 보드 2종류 추가, 폼을 보드로 흡수): 그리드 · 캘린더 · 칸반 · 간트(타임라인으로 이름 변경) · 피드(텍스트·긴텍스트·문서 렌더링 타입 중 1개가 있고 + 이미지 렌더링 타입이 있는 데이터 그룹만) · 갤러리(데이터 그룹이 아닌 파일 그룹을 기반으로 동작, 파일 그룹을 선택하면 해당 내용을 갤러리 형태로 보여줌) · 폼
5. **차트 위젯** (기존 대시보드 뷰 분해, 종류는 유지, 차트는 쓰기가 없는 읽기 전용 위젯): 막대 · 꺾은선 · 파이 · 산점도 · 카드 · 게이지 · 멤버
6. **파일 위젯**: 파일 업로더(파일 그룹 기반) · 상세 파일 업로더(파일 그룹 기반)
7. **탐색기 위젯** (탐색기는 쓰기를 포함하는 위젯): 데이터 탐색기 · 파일 탐색기(파일 그룹 기반)

## B. 렌더링 타입 (기존 "기능"을 "블록"으로 대체, 총 41개)

1. **문자 블록 (10개)**: 텍스트 `text`(str) · 긴텍스트 `longText`(str) · 숫자 `number`(num) · 링크 `link`(str) · 전화번호 `phoneNumber`(str) · 이메일 `email`(str) · 통화 `currency`(num) · 백분율 `percentage`(num) · 코드 `code`(str) · 비밀번호 `password`(str)
2. **버튼 블록 (12개)**: 스위치 `switch`(bool) · 체크박스 `checkbox`(bool) · 기간 `period`(json) · 날짜 `date`(date) · 날짜및시간 `dateTime`(date) · 시간 `time`(date) · 마감일 `deadline`(json) · 세계시간 `worldTime`(str) · 단일선택 `singleSelection`(json) · 복수선택 `multiSelection`(json) · 국가 `nation`(str) · 색상 `color`(json)
3. **특수 블록 (13개)**: 사용자 `user`(json) · 고유ID `uniqueID`(json) · 행ID `rowID`(num) · 태그 `tag`(json) · 투표 `vote`(json) · 평점 `rating`(json) · 진행률 `progress`(json) · 타이머 `timer`(num) · 수식 `formula`(json) · 빠른수식 `quickFormula`(json) · 자동화 `automation`(json) · 로그 `log`(json) · 마지막업데이트 `lastUpdate`(json)
4. **확장 블록 (6개)**: 연결 `connection`(json) · 행연결 `rowConnection`(json) · 듀얼블록 `dualBlock`(json) · 문서 `document`(json) · 이미지 `image`(json) · 파일 `file`(json)

> **★ 41 영문ID 정본 (단일 출처 — 마스터 확정·byte verbatim)**: 위 `englishID`가 **D8 `block_type_enum`(라벨 byte-exact)·R16 P1 fs-api `ALL_BLOCK_TYPE_IDS`·⑪ 드리프트체크**의 단일 정본. ⚠️ `button` 없음(분해→`automation`)·`uniqueID`/`rowID` 대문자 ID·`quickFormula`(≠simpleFormula)·`dualBlock`(≠dualWidget)·`rowConnection`(≠rowLink). dataType 분포 = **str9·num5·date3·bool2·json22**(timer=num·uniqueID/color=json — §F 매핑정정 반영).

## C. 렌더링 타입(위젯)의 동작 방식 — 시나리오 1번 (기본) (마스터 요구 원문)

- a. 사용자가 페이지를 생성한다.
- b. 페이지에 자동으로 섹션이 할당되어 있고 기본 옵션으로 높이 700px에 세로 확장 옵션이 켜져있다.
- c. 완전히 빈 섹션에서는 중앙에 "빈 섹션", "위젯을 추가하려면 헤더의 + 버튼 클릭" 안내 텍스트를 표기하고, 그 아래에 7가지 위젯 바로가기 아이콘 버튼을 서로 다른 배경색을 넣어서 모서리가 둥근 정사각형으로 표기한다.
- d. 사용자가 헤더 말풍선 내부의 + 아이콘 버튼을 누르면 모달이 나타나고, 위젯 종류 선택지가 나타난다. (모든 격자가 다 차있는 경우에는 보여주되, 비활성화됨)
- e. 위젯 종류(예: 보드 위젯)를 선택했다면, 보드 위젯 목록으로 넘어간다. (섹션의 바로가기 아이콘 버튼을 누른 경우 바로 이 단계로 오게 됨)
- f. 위젯을 선택했다면 상세 위젯 설정 및 미리보기 페이지로 넘어간다.
- g. 해당 페이지에서 모든 옵션은 데이터 그룹을 생성 또는 기존 데이터 그룹 선택을 하기 전까지 보여지지만 비활성 처리되어 조작 불가능. (예외: 그룹이 필요 없는 경우 스킵, 파일 그룹을 사용하는 경우 파일 그룹이 이 역할)
- h. 그룹 지정 후 열까지 지정 또는 생성해야 하는 위젯인 경우(블록 위젯처럼) 열까지 지정 또는 생성해야 옵션이 활성화된다.
- i. 각 위젯별 필요한 설정을 마치고 등록 버튼을 누르면 위젯이 등록된다. 이때 같은 부분을 건드리는 다른 계정이 있을 수 있기 때문에 최소 장치로 등록 직전에 DB를 조회해서 배치 가능 여부를 확인하고 배치한다.
- j. 배치된 위젯은 디자인 모드에서 평상시에는 뷰 모드와 동일하게 보여준다. (사용자가 뷰 모드로 바꾸지 않아도 디자인을 볼 수 있게) 하지만 마우스가 호버되면 디자인 모드에서 해당 위젯은 "하얀색" 불투명 막이 씌워지고 해당 막에는 드래그&드롭 아이콘 버튼(좌측 상단)과 설정 버튼(우측 상단)이 나타난다.
- k. 격자 옵션은 사용자가 선택한 위젯 종류와 세부 위젯 종류에 따라 기본적으로 지정해준다. (그리드는 기본 크게, 블록은 기본 작게 등; 기본 지정 크기가 남은 공간에서 불가능한 경우 현재 가능한 가장 큰 공간의 규격을 가져온다.)

## D. 추가 합의 사항 (마스터 요구 원문)

1. 모든 위젯(차트 포함, 파일 그룹 포함)은 페이지 관리 주기 설정의 영향을 받아야 한다. (위젯별 관리주기 옵션이 필수, 단일 데이터로 볼지 페이지 관리주기 연동할지, 해당 페이지가 관리주기 비활성 상태시에는 해당 옵션이 단일 데이터 고정임) → **관리주기 동작 기준 = 데이터 밸류의 `created_at`**(켜진 위젯 = 현재 주기 창에 드는 created_at 셀만 표시; R15⑩ BE 시간 필터 키 — 2026-06-26 마스터 확인).
2. 신규 구조에서 섹션은 한 페이지에서 여러개 필요가 없다, 높이(또는 자동 확장, 최소 높이 200)만 지정해주면 됨 (배경 디자인 역시 위젯이 처리 가능, 격자 열 자체를 폐기, 격자를 가로 100개·세로 70개로 고정하는 방식으로 변경)
3. 따라서 이제 섹션 추가, 삭제 자체가 없음 → 모든 페이지는 반드시 섹션 1개를 자동으로 가져야 함
4. 섹션 컨트롤바는 이제 위젯 추가, 높이 입력, 세로 확장 옵션만 가지기 때문에 기존 플로팅 컨트롤 바 방식이 과해짐 → 따라서 제거하고 해당 3개 기능을 디자인 모드에서만 헤더에 나타나는 말풍선 아이콘 박스 내부로 이동
5. 말풍선 아이콘 박스에는 나만의 디자인 바로가기 아이콘 버튼도 있어야 함 → 이게 설계 핵심 중 하나인 위젯 설정 즐겨찾기임 → 여기서 즐겨찾기 위젯 설정 목록을 조회하거나 새로운 즐겨찾기 설정을 만들 수 있음
6. 설정 → 사용자 설정의 동작 방식이 대폭 변경 됨 → 이제는 해당 위치에서 폼을 등록하는게 아니라 즐겨찾기 항목 중 최상위 위젯에 해당하는 것을 등록하고 보여줌 (권한 처리도 기존 철학과 유사하게 가면 됨)
7. 이제 디자인 모드에서 모든 위젯은 바로 설정 불가능 → 모든 설정은 마우스 호버 후 설정을 눌러 나타나는 모달에서 처리함 (특히 기존 데이터 뷰어의 그리드 뷰→열 메뉴 같은 부분들이 전부 이동되야 함)
8. DB에 41개 블록 타입으로 enum을 신설하고 data_column 테이블에 "블록 타입" 열(Not Null)을 추가한다. → 모든 위젯이 열을 생성(자동 생성, 수동 생성 모두)할 때 반드시 사용자가 렌더링 타입(블록 타입)을 지정하는 과정이 있어야한다.
9. 탐색기 위젯은 이제 뷰어, 서브, 폼, 일반으로 구분해서 보여지지 않는다. → 설계에 따르면 모든 데이터 그룹이 동등해지기 때문에 하나로 탭 없이 보여주면 된다.

## E. 추가 합의 (위젯 설정 모달 / 배경)

- 기존에 위젯이 자체적으로 가지는 **모든 디자인 모드 설정**과 **위젯 설정 드로어가 제공하는 항목**이 전부 **새로운 위젯 설정 모달로 이동**.
- 위젯 설정 모달 = **3개 탭**: ① 위젯 설정 ② 위젯 디자인 ③ 배경 디자인(격자가 이곳에 위치).
- **배경 디자인 = 위젯별 배경**(공유 아님). 위젯이 가지는 격자 내부에서만 적용.

## F. 데이터 모델 확정 (크럭스 해소)

- **블록타입은 열당 1개 고정**(위젯별 가변 아님). `data_column.블록타입`(Not Null, 41 enum)이 단일 출처.
- 위젯 설정 모달 *위젯 설정 탭*이 신규 열 생성 시 블록타입을 지정 → `data_column`에 저장.
- **표현 전환 = "열 타입 변경(마이그레이션)" 기능**으로만. 셀 데이터(5타입)는 그대로, 블록타입만 변경 → 그 열을 쓰는 **모든 위젯이 자동 전환**.
- **호환 제약 (2026-06-26 적대검증 정정)**: 열 타입 변경은 **2층 호환(dataType + shape)** 범위 내에서만. ~~"같은 데이터타입 내 = 데이터 무변경 즉시"~~ 는 **거짓** — json 22블록은 shape가 상이(single→multi=스칼라→배열·user/document/vote 제각각)하고 `conversion-catalog.listTargetTypes`가 json 22중 13(log·lastUpdate·formula·quickFormula·uniqueID·image·file·vote·rating·progress·connection·rowConnection·dualBlock)을 변환소스서 제외. 따라서 **shape 호환표(2층)가 마이그 기능의 선행 필수**(dataType 1층만으론 사일런트 셀손상). `listTargetTypes` + `column_type↔viewer_type 불변식`이 경계 가드. (R16 P1.5.)
- **blockType→dataType 매핑 (2026-06-26 마스터 확정)**: 분포 **string9/number5/date3/boolean2/json22**. **timer=number**(초·FE 계산, 셀 "MM:SS"→초)·uniqueID=json(열ID+행ID+그룹ID 조합 구조)·color=json(확장성). 정본=계약(R16 P1)·DB enum(R15 D8)·교차repo 드리프트체크(R15⑪).
- **파생/계산 블록 7종**(rowID·uniqueID·lastUpdate·log·formula·quickFormula·dualBlock)=셀값 미저장 → dataType 명목·셀 마이그 비대상(**가상/파생 열 개념** — D8 백필·마이그가 계산열을 변환데이터로 오인 방지).
- **"열 1개 = 블록타입 1개"는 뷰모드별 다른 표현을 막지 않음**(같은 status 열을 그리드=셀, 칸반=그룹핑 축으로 쓰는 건 뷰모드 차이지 블록타입 차이가 아님).
- = Notion 모델(속성 타입 1개, 변경 시 전체 반영)과 동형.

## G. 로드맵 영향 (검토 — main 세션 분석)

> 마스터 판단: "DB 로드맵은 일부 조정/개선, FE 로드맵은 전면 재개편" — **둘 다 타당.** 근거·뉘앙스 ↓

### DB(Roadmap-15) = 조정/개선 (재작성 아님)
- R15의 데이터층 재설계(5 데이터타입·JSON 셀·신규 테이블 permission/company_setting/widget_preset/data_row·SP 비-삼킴·value_data→data)는 **본 설계의 올바른 토대**. 진행 중인 R15 작업(흡수 M1/M2 완료, SP dual-write 진행)은 **무효화되지 않음**.
- 본 FE 설계가 추가로 요구하는 DB 변경은 **경계적 조정**:
  - **D8**: `data_column.블록타입`(Not Null) + 41-블록 enum 신설 + 기존 type 컬럼 → 블록타입 **백필**. (기존 type 컬럼 정련 수준; task-db-structure 소관.)
  - **+적대검증(2026-06-26) 도출 — '한정적 추가'보다 큼**: 신 모델이 완료 R15 작업과 **충돌**하는 정합 단위 — company_setting 폼락(`widget_type='form'` 하드락) 해제·page_widget.preset_id 참조계약 부재·page_section/정수격자 미편성(area→격자 마이그). + 신규 BE: widget_preset 3계층·recent-usage 조회·file-group·D1·교차repo 드리프트체크.
- → R15는 **유지·정련 + BE 슬라이스 8단위(④~⑪) 확장**(전부 additive·i-dev 머지·컷오버 합류). D8·④~⑪는 task-db-structure/task-db-data/plan-enterprise로 합류.

### FE(Roadmap-16) = 전면 재개편 (단, 완료 토대 위에)
- 본 설계는 "view-mode 팬아웃"을 훨씬 넘는 **위젯/페이지 시스템 전면 재설계**(블록 위젯·보드/차트/파일/탐색기 위젯·41 블록·페이지/섹션 모델·3탭 설정 모달·즐겨찾기·파일그룹·관리주기). 기존 Roadmap-16의 R16②/③ 틀은 **재작성** 필요.
- **그러나 이미 완료한 작업은 토대로 재사용**(폐기 아님):
  - R16① 뷰어 통합/단일 패키지 → 보드 위젯(그리드/캘린더/칸반/타임라인)의 통일 엔진.
  - sub/external 4패키지 삭제 → D9(탐색기 통일·데이터그룹 동등)와 정합 = 사후 정당화.
  - R16② P1 렌더링타입 레지스트리(P1a/P1b) → 41-블록 렌더링 시스템의 토대(패턴 유지, **목록·분류를 41-블록[문자/버튼/특수/확장]으로 재정렬** + 데이터타입↔블록타입 호환표).
- → FE 로드맵의 **전방 계획(프롬프트 시퀀스)을 재작성**하되, 완료된 R16① + 레지스트리 토대 + 단일 패키지 위에 쌓음.
